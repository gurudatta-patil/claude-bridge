# ts-rs Export Guide for Stitch Rust Sidecars

`ts-rs` is a Rust derive macro that generates TypeScript type definitions at compile time directly from your Rust structs. In a Stitch sidecar this means your Rust source file is the single source of truth: change a field in Rust and the TypeScript bindings update on the next `cargo test` run — no separate codegen script, no schema file to keep in sync.

---

## 1. Adding ts-rs to Cargo.toml

Open `languages/rust/Cargo.toml` (or whichever crate hosts your sidecar types) and add:

```toml
[dependencies]
serde        = { version = "1", features = ["derive"] }
serde_json   = "1"
ts-rs        = { version = "7", features = ["serde-compat"] }
```

The `serde-compat` feature tells `ts-rs` to honour your `#[serde(...)]` attributes (e.g. `rename_all`, `rename`, `skip`) when generating TypeScript names, so the emitted types match what actually flows over the wire.

---

## 2. Annotating Request/Response Structs

Import the derive macro and add `#[ts(export)]` to every struct you want exported.

```rust
use serde::{Deserialize, Serialize};
use ts_rs::TS;

/// Parameters sent from the TypeScript host to the Rust sidecar.
#[derive(Debug, Serialize, Deserialize, TS)]
#[ts(export)]
pub struct AddParams {
    pub a: f64,
    pub b: f64,
}

/// Response returned from the Rust sidecar to the TypeScript host.
#[derive(Debug, Serialize, Deserialize, TS)]
#[ts(export)]
pub struct AddResult {
    pub result: f64,
}

/// Example with optional fields and renaming.
#[derive(Debug, Serialize, Deserialize, TS)]
#[serde(rename_all = "camelCase")]
#[ts(export)]
pub struct UserRecord {
    pub user_id: u64,
    pub display_name: String,
    pub email: Option<String>,
}
```

By default `ts-rs` writes generated files into a `bindings/` directory at the crate root. You can override the output path per-struct:

```rust
#[ts(export, export_to = "../../languages/typescript/src/bindings/")]
pub struct AddParams { ... }
```

Or set a crate-wide default in `Cargo.toml`:

```toml
[package.metadata.ts-rs]
export_dir = "../../languages/typescript/src/bindings"
```

## 3. Exporting via `cargo test`

`ts-rs` registers a hidden test for every `#[ts(export)]` struct. Run the test suite to trigger generation:

```bash
cd languages/rust
cargo test export_bindings
```

This creates (or overwrites) one `.ts` file per exported type:

```
languages/rust/bindings/
  AddParams.ts
  AddResult.ts
  UserRecord.ts
```

Each file looks like:

```typescript
// This file was generated by [ts-rs](https://github.com/Aleph-Alpha/ts-rs).
// Do not edit this file manually.

export type AddParams = { a: number; b: number; };
```

To regenerate all bindings in one shot (useful in CI):

```bash
cargo test
```

The `export_bindings` tests run as part of the normal test suite, so `cargo test` always keeps them current.

## 4. Using the Generated Types in the TypeScript Client

After generation, import the types directly into the Stitch TypeScript host:

```typescript
// languages/typescript/src/index.ts
import type { AddParams, AddResult } from "./bindings/AddParams";
import type { UserRecord }           from "./bindings/UserRecord";
import { bridge }                    from "./bridge";   // your Stitch bridge instance

async function add(a: number, b: number): Promise<AddResult> {
  const params: AddParams = { a, b };
  return bridge.call<AddResult>("add", params);
}

async function getUser(id: number): Promise<UserRecord> {
  return bridge.call<UserRecord>("getUser", { user_id: id });
}
```

Because the TypeScript type and the Rust struct share the same definition you get end-to-end type safety: if you add a required field in Rust, the TypeScript side will produce a compile error until you update the call sites.

## 5. Automating with a Build Script

To regenerate bindings on every `cargo build` rather than only on `cargo test`, add a `build.rs` at the crate root:

```rust
// languages/rust/build.rs
fn main() {
    // Tell Cargo to re-run this script whenever a source file changes.
    println!("cargo:rerun-if-changed=src/");
}
```

Then trigger the export inside the build script using `ts_rs::export!`:

```rust
// build.rs
use std::path::PathBuf;

fn main() {
    println!("cargo:rerun-if-changed=src/");

    // Collect all exported types and write them in one pass.
    // Requires ts-rs ≥ 7.
    let out_dir = PathBuf::from("../../languages/typescript/src/bindings");
    std::fs::create_dir_all(&out_dir).unwrap();

    ts_rs::export! {
        crate::AddParams   => out_dir.join("AddParams.ts"),
        crate::AddResult   => out_dir.join("AddResult.ts"),
        crate::UserRecord  => out_dir.join("UserRecord.ts"),
    }
}
```

> Note: `ts_rs::export!` requires the types to be in scope, so import them at the top of `build.rs`. Alternatively keep using the test-driven approach (Section 3) and invoke `cargo test` in your CI pipeline before the TypeScript build step.

## 6. Integration with the Stitch TypeScript Client Template

### Where to put the generated files

Place generated bindings under the TypeScript client's source tree so `tsc` picks them up automatically:

```
languages/typescript/
  src/
    bindings/          ← ts-rs output lands here
      AddParams.ts
      AddResult.ts
      UserRecord.ts
    bridge.ts          ← Stitch bridge wrapper
    index.ts           ← your application code
  tsconfig.json
  package.json
```

Add the `bindings/` directory to `.gitignore` if you prefer to treat generated files as build artefacts, or commit them if you want the TypeScript side to be buildable without a local Rust toolchain.

```
# .gitignore (TypeScript client)
src/bindings/
```

### tsconfig.json — no changes needed

`ts-rs` emits plain `.ts` files that use only `export type`, so they work with every `moduleResolution` setting (`node`, `bundler`, `node16`, etc.) without modification.

### Recommended CI pipeline order

```yaml
# .github/workflows/ci.yml (excerpt)
steps:
  - name: Generate TypeScript bindings from Rust
    working-directory: languages/rust
    run: cargo test export_bindings

  - name: Build TypeScript client
    working-directory: languages/typescript
    run: |
      npm ci
      npm run build    # tsc picks up freshly generated bindings
```

This guarantees the TypeScript build always sees bindings that match the current Rust source.

## Quick Reference

| Task | Command |
|------|---------|
| Generate all bindings | `cargo test export_bindings` |
| Generate + run all tests | `cargo test` |
| Change output directory (per struct) | `#[ts(export_to = "path/")]` |
| Change output directory (crate-wide) | `[package.metadata.ts-rs] export_dir = "path"` in `Cargo.toml` |
| Rename a field in TypeScript only | `#[ts(rename = "newName")]` on the field |
| Skip a field | `#[ts(skip)]` on the field |
| Inline a nested type | `#[ts(inline)]` on the field |
